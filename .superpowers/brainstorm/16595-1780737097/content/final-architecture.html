基于真实前端代码的精简集成方案

关键发现：前端不区分 goal/confirm/replan — 全部走 execute()

当前消息流

→

修正后的集成方案

ChatWebSocketHandler 改动（最小化）
唯一改动：当 ctx.confirmedPlan != null 时，将后续消息路由到 confirm() 
// 修改前（永远走 execute）
} else {
    chatService.execute(ctx, content, deepThinking) ...

// 修改后（确认后的消息走 confirm）
} else if (ctx.confirmedPlan != null) {
    chatService.confirm(ctx, content) ...  // ← 新增路由
} else {
    chatService.execute(ctx, content, deepThinking) ...
}

对 confirm() 的影响

AgentLoop.confirm() — 自动重试 + 继续执行
SUCCESS
（含 LONG_WAIT）
→ 继续下一条 Tool
→ DONE 汇总等待信息
FULLY_BOOKED
NOT_AVAILABLE
→ while 循环内重新调 LLM
→ ≤3 次，逐步扩大搜索半径
超过 3 次
→ ERROR: "自动调整失败"
→ 用户可手动重新规划
