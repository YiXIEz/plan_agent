AgentLoop.confirm() 流程重构

现有计划中 while 循环的三分支行为

计划中的 confirm() 流程
LLM 调用
生成 ToolCall 列表
逐条执行 Tool
toolRegistry.execute()
ReplanningHandler.analyze()
判断 FailureType
✅ SUCCESS
继续下一条 Tool
⏳ LONG_WAIT
sink.next(NOTIFY)
sink.complete() ← 终止
剩余 Tool 未执行!
❌ FAILED
replanCount++
break → while 重新调 LLM

核心问题：LONG_WAIT 后的状态恢复

用户对 NOTIFY 做出选择后，如何继续执行剩余的工具？

A

方案 A: confirm() 支持 "暂停-恢复"

LONG_WAIT 时不调用 sink.complete()，而是暂停等待。用户在 WebSocket 同一会话中回复 "accept" 或 "replace"，AgentLoop 从暂停点恢复执行剩余 Tool。

✅ 状态不丢失   ⚠️ 需要暂停机制（Project Reactor 的 Flux.create 不原生支持"等待外部事件后继续"）

B

方案 B: confirm() 终止，replan() 接管全部后续

LONG_WAIT 时 confirm() 结束并记录"未完成列表"。用户回复后，replan() 从断点继续执行剩余 Tool（accept）或替换后执行（replace）。

✅ 流程清晰   ⚠️ SessionContext 需要记录 "待执行 Tool 列表"

C

方案 C: 不暂停，NOTIFY 仅作信息提示

LONG_WAIT 时发送 NOTIFY 但不停下——继续执行后续 Tool。最后汇总所有 NOTIFY 给用户一次性决策。适合 所有预订都执行完再统一调整 的模式。

✅ 实现最简单   ⚠️ 用户可能不需要等待的通知项的上下文来决策
